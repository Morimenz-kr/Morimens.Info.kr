# 캐릭터 데이터 파일 구조 메모

이 문서는 향후 캐릭터 데이터 리팩토링 때 참고하기 위한 메모다.

## 배경

현재 캐릭터의 스킬, 영지 각성, 최종 법칙, 계령 정보는 주로 `data/character_effects.json`에 모여 있다.

특성 정보는 아직 이 데이터셋에 공통 필드로 정리되어 있지 않다. 앞으로 특성을 추가하거나 캐릭터 데이터 범위를 넓힐 경우, 하나의 거대한 JSON 파일을 계속 유지할지, 캐릭터별 JSON으로 나눌지 판단이 필요하다.

## 선택지 1: 단일 JSON 유지

예시:

```txt
data/character_effects.json
```

장점:

- 전체 캐릭터 데이터를 한 번에 불러오기 쉽다.
- 캐릭터 비교, 검색, 필터링 기능에서 fetch 횟수가 적다.
- 정적 사이트 구조에서는 구현이 단순하다.
- 현재 코드와 가장 잘 맞는다.

단점:

- 파일이 계속 커지면 사람이 수정하기 어렵다.
- 특정 캐릭터만 수정해도 큰 파일 diff가 발생한다.
- 여러 캐릭터를 동시에 갱신할 때 충돌 가능성이 커진다.
- 스킬, 계령, 특성, 추천 세팅, 출처 같은 정보가 늘어나면 구조가 둔해질 수 있다.

## 선택지 2: 캐릭터별 JSON 분리

예시:

```txt
data/characters/24.json
data/characters/tawil.json
data/characters/pontos.json
```

예상 구조:

```json
{
  "id": "pontos",
  "name": "폰토스",
  "skills": [],
  "breakthroughs": [],
  "traits": [],
  "sources": []
}
```

장점:

- 한 캐릭터의 정보를 한 파일에서 관리할 수 있다.
- 새 캐릭터 추가와 수정 diff가 작아진다.
- 특성, 추천 세팅, 출처, 업데이트 날짜 같은 정보를 확장하기 쉽다.
- 장기적으로 유지보수와 검증 스크립트 작성이 편해진다.

단점:

- 현재 `character_effects.json` 하나를 읽는 코드와 맞지 않는다.
- 목록/검색 화면에서 여러 파일을 fetch해야 할 수 있다.
- 캐릭터 목록용 manifest 또는 인덱스 파일이 필요하다.
- 마이그레이션 시 기존 화면 로직을 함께 수정해야 한다.

## 추천 방향

단기적으로는 `data/character_effects.json`에 `traits` 같은 공통 필드를 추가해서 현재 구조 안에서 특성을 먼저 완성한다.

장기적으로는 원본 데이터를 캐릭터별 JSON으로 관리하고, 사이트에서 사용할 배포용 JSON은 빌드/검증 스크립트로 합치는 방식을 고려한다.

예시:

```txt
data/characters/*.json          # 사람이 수정하는 원본
data/character_effects.json     # 사이트가 읽는 빌드 결과물 또는 호환용 파일
tools/build-character-effects.* # 원본을 합치는 스크립트
```

이 방식은 수정할 때는 작은 파일을 다루고, 웹에서는 기존처럼 한 번에 데이터를 읽을 수 있어서 절충안으로 좋다.

## 리팩토링 시 체크할 것

- 기존 화면이 `character_effects.json`을 직접 fetch하는지 확인한다.
- 캐릭터 ID 기준의 manifest가 충분한지 확인한다.
- 특성 필드 이름은 `traits`처럼 기존 `skills`, `breakthroughs`와 같은 영어 키로 맞추는 편이 좋다.
- 출처가 필요한 경우 캐릭터별 `sources` 또는 항목별 `source` 필드를 둘지 결정한다.
- 마이그레이션 후 JSON 파싱과 누락 검증 스크립트를 반드시 추가한다.
